Your network watchtower.

Practical security visibility for the networks you actually run.

Vedetta is a self-hosted watchtower for homelabs, technical home users, small businesses, and hands-on operators. Core runs in Docker, the native sensor handles local visibility, and the current product is strongest at device discovery, passive DNS visibility, and local threat scoring.

You can get to useful local visibility quickly, but the current path still assumes Docker, a native sensor, local network access, and some comfort with sudo. Pi-hole and AdGuard Home remain optional integrations, not the product identity.

Ground Truth

What a visitor should understand in under 20 seconds

  • Vedetta is self-hosted and local-first.
  • Core runs in Docker. A native sensor provides the strongest current visibility.
  • Device discovery and passive DNS are today’s strongest visibility layers.
  • Pi-hole and AdGuard Home are optional sources, not requirements.
  • Router and firewall aggregation is part of the next expansion.
What Vedetta Is Today

A practical network watchtower, not a pretend appliance.

The strongest current value is lightweight, self-hosted visibility. Vedetta is already useful locally without a cloud dependency, and it gets richer when you connect more signal sources over time.

Today

Device discovery and inventory

Vedetta uses a native sensor to discover what is on your network, keep a device inventory, and surface new or changed devices quickly.

Today

Passive DNS and local network signals

The current sensor captures DNS activity directly and complements it with ARP, DHCP, mDNS, and SSDP/UPnP observations for a practical local detection workflow.

Today

Local threat scoring

Vedetta already scores DGA-like domains, beaconing, tunneling, rebinding, and DNS bypass patterns, then enriches them with local threat intelligence.

Optional

Pi-hole and AdGuard Home integrations

If you already run Pi-hole or AdGuard Home, Vedetta can pull value from those logs too. They are integrations, not prerequisites.

Expansion

Router and firewall coverage

Early connector groundwork exists in the repo, with UniFi work started and OpenWRT, pfSense/OPNsense, and MikroTik on the near-term expansion path.

Trust

Local ownership first

The local deployment is the product. Telemetry is optional, and the future community threat network should stay opt-in and privacy-conscious.

Status

Clear lines between shipped work and roadmap work.

Vedetta is in active development. These buckets are the honest version of the product state right now.

Available now

Local monitoring that already stands on its own

  • Docker-based Core with API, UI, and local storage
  • Native sensor for macOS and Linux workflows
  • Passive DNS capture plus active and passive device discovery
  • Local DNS scoring and enrichment
  • Optional Pi-hole and AdGuard Home pollers
In progress

Sharpening the alpha product

  • Install and onboarding polish
  • Token rotation and broader admin/dashboard auth hardening
  • More durable docs and setup guidance
  • Turning early connector code into testable workflows
Planned next

Broader network visibility layers

  • Router and firewall log aggregation
  • Connectors for UniFi, OpenWRT, pfSense/OPNsense, and MikroTik
  • Better correlation and multi-network handling across ARP, DHCP, mDNS, and SSDP/UPnP signals
  • An optional, privacy-conscious community threat network
Who It Is For

Grounded fit, honest install, explicit trust model.

Vedetta is being built for small environments that want real visibility without pretending they already have a full SOC.

Audience

Who Vedetta fits today

  • Homelab users
  • Technical home users
  • Small businesses without a full security team
  • Consultants, MSPs, and security practitioners helping tiny environments
Required vs optional

Core plus sensor is the current model

Required today: Vedetta Core and at least one native sensor. Optional today: Pi-hole, AdGuard Home, telemetry, and early connector experimentation.

Install reality

Alpha, homelab-friendly, not plug-and-play

The current path uses Docker for Core and a native sensor install with elevated local access. That is honest alpha software, not a fake frictionless setup story.

Privacy and trust

Self-hosted first

The local deployment should stay useful with no cloud dependency. Telemetry is optional, off by default, and the community intelligence layer is future-facing.

Pi-hole positioning

Not a Pi-hole companion product

Pi-hole and AdGuard Home are useful if you already run them. Vedetta is being built to ingest useful security signals from multiple DNS and network data sources over time.

Security note

Local-first alpha means explicit boundaries

Machine-to-machine sensor auth is in place. Token rotation and broader admin/dashboard hardening are still ahead, and the current product should still be treated as LAN-first alpha software, not hardened public internet infrastructure.

Start

Two moving parts, by design.

Core is the dashboard and data plane. The sensor is the local network-side view. That split is where the current product gets its value.

1. Core

Start Vedetta Core

Core runs locally with Docker Compose.

git clone https://github.com/MahdiHedhli/vedetta.git
cd vedetta
docker compose up -d
2. Sensor

Install the native sensor

The current public installer supports macOS and Linux. Review the script before running it.

curl -fsSL -o /tmp/vedetta-sensor-install.sh \
  https://raw.githubusercontent.com/MahdiHedhli/vedetta/main/sensor/deploy/install.sh

sudo bash /tmp/vedetta-sensor-install.sh \
  --core http://<CORE_IP>:8080

If you want the longer operational details, the repo README and roadmap go deeper on what is available now, what is optional, and what is still next.
